02 · Schema

The table, the indexes, the guarantees

Everything lives here. Indexes are designed for the dashboard's query pattern: filter by event_name + date range + exclude demo/admin.

CREATE TABLE
CREATE TABLE analytics_events (
  id              bigserial PRIMARY KEY,
  event_name      text NOT NULL,
  user_id         uuid REFERENCES surfers(id) ON DELETE CASCADE,
  conversation_id uuid REFERENCES conversations(id) ON DELETE CASCADE,
  occurred_at     timestamptz NOT NULL DEFAULT now(),
  properties      jsonb,
  is_demo_user    boolean NOT NULL DEFAULT false,
  is_admin        boolean NOT NULL DEFAULT false
);
Main dashboard index
CREATE INDEX idx_events_name_time
  ON analytics_events (event_name, occurred_at)
  WHERE NOT is_demo_user AND NOT is_admin;
"Once per user" uniqueness (milestones)
CREATE UNIQUE INDEX idx_events_first_time
  ON analytics_events (user_id, event_name)
  WHERE event_name IN ('user_signed_up',
    'onboarding_step_1', …, 'onboarding_step_8',
    'onboarding_finalized',
    'swelly_search_clicked',
    'swelly_connect_clicked',
    'first_message_sent');
03 · Date filter

One range control applied to everything

The admin picks the range once (preset or custom). All tiles, funnels, and sparklines recompute with that range.

Range: Today 7d 30d 90d 1y All time Custom: May 1 → May 18 ↳ applies to everything below
Single query pattern (any metric)
SELECT COUNT(*) FROM analytics_events
WHERE event_name = $1
  AND occurred_at >= COALESCE($from, '-infinity')
  AND occurred_at <  COALESCE($to,   'infinity')
  AND NOT is_demo_user
  AND NOT is_admin;
04 · The 9 events · card by card

Where it's stored · when it updates · how it's used

Each metric = one event_name. One row per user for milestones; many rows for sessions.

user_signed_up
Signup
When it updates
Right after a successful signup (auth flow finished).
Where it's written
src/services/auth/authService.ts → post-signup handler 
logEvent('user_signed_up', { userId })
Resulting row
{ event_name: 'user_signed_up',
  user_id: '…',
  occurred_at: now(),
  is_demo_user, is_admin }
How the dashboard uses it
Hero KPI "Users created" · first step of the onboarding funnel.
onboarding_step_1 … onboarding_step_8
Client
When it updates
When the user taps "Next" on each of the 8 onboarding steps. Only the first time the user completes that step (idempotent).
Where it's written
src/screens/OnboardingStep1Screen.tsx … OnboardingStep4Screen.tsx (+ video upload + budget + lifestyle) 
// in the "Next" handler
logEvent('onboarding_step_N', { userId })
Uniqueness guarantee
Unique index on (user_id, event_name) with ON CONFLICT DO NOTHING. If the user goes back and passes again, no duplicate is created.
How the dashboard uses it
Onboarding funnel — 1 row per step, showing how many users reached it and where they dropped off.
onboarding_finalized
Client
When it updates
When the user taps the final "Got it" / "Save" button of the full onboarding.
Where it's written
Final onboarding screen → "Got it" button handler 
logEvent('onboarding_finalized', { userId })
// also: surfers.finished_onboarding = true
How the dashboard uses it
Hero KPI "Onboarding completed" · last step of the funnel.
swelly_search_clicked
Client
When it updates
First time the user enters the Swelly chat (tapped "find surfers"). Once per user.
Where it's written
src/screens/TripPlanningChatScreen.tsx (~line 263, next to trackSwellyChatEntered) 
logEvent('swelly_search_clicked', { userId })
How the dashboard uses it
Tile "Clicked Swelly search" · step 1 of the engagement funnel.
swelly_connect_clicked
Client
When it updates
First time the user taps "Connect" / "Message" on a Swelly match card.
Where it's written
src/components/MatchedUserCard.tsx → Connect button handler 
logEvent('swelly_connect_clicked', {
  userId,
  properties: { target_user_id }
})
How the dashboard uses it
Tile "Made a Swelly match" · step 2 of the engagement funnel.
first_message_sent
Trigger
When it updates
First time the user inserts a row into messages (not system, not deleted).
Where it's written
Postgres trigger: on_message_insert() AFTER INSERT ON messages 
INSERT INTO analytics_events
  (event_name, user_id, conversation_id, occurred_at, …)
VALUES ('first_message_sent', NEW.sender_id, …)
ON CONFLICT (user_id, event_name) DO NOTHING;
How the dashboard uses it
Tile "Sent first message" · step 3 of the engagement funnel.
conversation_two_sided
Conversation
When it updates
When the second DM participant sends their first message (both sides now have ≥1).
Where it's written
Same on_message_insert() trigger on messages 
// user_id NULL · conversation_id set
// SKIPPED if any participant is demo or admin
INSERT INTO analytics_events
  (event_name, conversation_id, occurred_at)
VALUES ('conversation_two_sided', NEW.conversation_id, …);
How the dashboard uses it
Tile "Matches with reply" · step 4 of the engagement funnel. Already pre-filtered for demo/admin at write time, no extra check needed.
conversation_deep_engaged
Conversation
When it updates
When both participants reach ≥4 messages each in the same conversation.
Where it's written
Same on_message_insert() trigger — "deep engaged" block 
IF v_my_msg_count >= 4 AND v_their_msg_count >= 4 THEN
  INSERT INTO analytics_events
    (event_name, conversation_id, occurred_at)
  VALUES ('conversation_deep_engaged', NEW.conversation_id, …);
END IF;
How the dashboard uses it
Tile "4+ msg each side" · last step of the engagement funnel.
app_opened
Session
When it updates
Every time the app boots (not just the first). Throttle: max 1 insert per 30 min per user, controlled via AsyncStorage.
Where it's written
src/components/AppContent.tsx → useEffect on mount, throttled 
const last = await AsyncStorage.getItem('last_app_open_logged');
if (!last || Date.now() - +last > 30 * 60 * 1000) {
  logEvent('app_opened', {
    userId,
    properties: { platform, app_version }
  });
  await AsyncStorage.setItem('last_app_open_logged', String(Date.now()));
}
No unique index
It's the only event without a uniqueness constraint — it can repeat many times per user.
How the dashboard uses it
Tile "App opens (unique users)" with COUNT(DISTINCT user_id) over the selected range.
Replaces the old "Active users" tab. 
SELECT COUNT(DISTINCT user_id)
FROM analytics_events
WHERE event_name = 'app_opened'
  AND occurred_at BETWEEN $from AND $to
  AND NOT is_demo_user AND NOT is_admin;
