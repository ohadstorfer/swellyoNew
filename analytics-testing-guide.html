Group Trips — Analytics Testing Guide

Every action below writes one row to analytics_events. You do the action in the app, run the CHECK query, and confirm the row appeared. Same idea as the notifications guide — but here the “bell” is the analytics table, and the payoff is the Trips Analytics dashboard (retention · feature adoption · trip health).

The testers: Ohad (ohad.storfer@gmail.com · ecaaa678…) and Eyal (eyal@swellyo.com · e11a30c8…) — both admins (their rows are is_admin=true, hidden on the dashboard until the admin filter is off) — plus sababa (surfingram.contact@gmail.com · 845a0c7c…), a non-admin tester whose events show on the dashboard normally. Group trips are dev-only on prod, so testing on real trips is safe.

1How analytics events work (read once)

Where it lands
One table: analytics_events — columns event_name, user_id, trip_id, properties (jsonb), occurred_at, plus is_demo_user / is_admin set automatically by a DB trigger.
Who logs it
The person who did the action. A host approving a request logs as the host; a member committing logs as the member.
Fire-and-forget
Logging never blocks the UI and never shows an error. If it fails (or analytics is opted-out), the row just doesn’t appear.
The dashboard
Reads these rows through the analytics-trips edge function and builds 3 charts. It normally excludes admins — see the ⚠️ below.
⚠️ Ohad & Eyal are admins — the dashboard hides them by default. The analytics-trips function filters out is_admin = true rows, so your test events would never show on the charts. For this testing round the admin filter was temporarily removed in supabase/functions/analytics-trips/index.ts (marked TEMP DEBUG … RESTORE before prod). It only takes effect once that edge function is re-deployed (copy-paste into the Supabase dashboard). The CHECK query below shows the is_admin column directly, so you can verify rows land regardless of whether the function is re-deployed yet.
⚠️ Two events are throttled. trip_opened and trip_chat_opened log at most once per 30 min, per trip, per device (a “was active today” presence signal — see logEventThrottled). To re-test the same one quickly, use a different trip, wait 30 min, or clear the app’s AsyncStorage.

2The CHECK query

Run this in the Supabase SQL editor after each action. The newest row is on top.

select e.event_name,
       e.properties->>'action'   as action,
       coalesce(t.title,'—')       as trip,
       u.email                     as actor,
       e.is_admin, e.is_demo_user,
       to_char(e.occurred_at,'DD/MM HH24:MI:SS') as at_time
from analytics_events e
join users u on u.id = e.user_id
left join group_trips t on t.id = e.trip_id
where lower(u.email) in ('ohad.storfer@gmail.com','eyal@swellyo.com','surfingram.contact@gmail.com')
  and e.event_name like 'trip\_%'
order by e.occurred_at desc
limit 25;

How to read it:

You see…Meaning
A new top row with the expected event_name✅ The event was logged. Check action, trip and actor match what you did.
is_admin = trueNormal for Ohad/Eyal. The row exists; it just won’t show on the live dashboard until the admin filter is removed & the function re-deployed.
trip = — (null trip)Expected for a few events that don’t carry a trip id (trip_gear_claim, and the decline paths of gear/join). Still counts for the adoption chart (which keys on the user, not the trip).
No new rowEither analytics is opted-out in app settings, the action errored, or (for presence events) you’re inside the 30-min throttle window.

3Part 1 — Presence events throttled 30 min

These two power the trip-health and retention charts. Use a trip you haven’t opened in the last 30 minutes.

P1 — Open a trip either tester
  1. Open the detail screen of any trip (e.g. Costa waves or Maldives Surf n Fire).
  2. Run CHECK.
✅ trip_opened · trip = the one you opened · actor = you. No new row if you opened it <30 min ago.
P2 — Open the trip chat either tester
  1. Open the group chat of a trip you haven’t chatted in for 30 min.
  2. Run CHECK.
✅ trip_chat_opened · trip = that trip · actor = you.

4Part 2 — Host actions do as the host

Do these on a trip you host (Eyal hosts Trip 1 / Trip 2 / Costa waves / Maldives; Ohad hosts Ericeira Surf Week). The host is the one logged.

H1 — Post a trip update
  1. As host, post an update message on your trip.
  2. Run CHECK.
✅ trip_admin_update · trip set · actor = host. (no action)
H2 — Approve a join request
  1. Have the other tester request to join your trip, then approve it.
  2. Run CHECK.
✅ trip_join_decision · action = approve · actor = host.
H3 — Decline a join request
  1. Other tester requests to join; you decline.
  2. Run CHECK.
✅ trip_join_decision · action = decline · trip may show — (decline path doesn’t always carry the trip id).
H4 — Add a group-gear item
  1. As host, add an item to the trip’s group gear list (e.g. “Tent”).
  2. Run CHECK.
✅ trip_gear_added · action = add · trip set · actor = host.
H5 — Remove a group-gear item
  1. Delete a group-gear item.
  2. Run CHECK.
✅ trip_gear_added · action = remove.
H6 — Edit the packing-suggestion list
  1. As host, edit the host packing suggestions for the trip.
  2. Run CHECK.
✅ trip_gear_suggestion · trip set · actor = host. (no action)
H7 — Approve a member’s gear proposal
  1. Other tester proposes a gear item (test M4), then you approve it.
  2. Run CHECK.
✅ trip_gear_request · action = approve · actor = host.
H8 — Decline a member’s gear proposal
  1. Member proposes gear; you decline.
  2. Run CHECK.
✅ trip_gear_request · action = decline · trip may show —.
H9 — Approve a commitment
  1. Member submits a commit (test M5), then you approve it.
  2. Run CHECK.
✅ trip_commit · action = approve · actor = host.
H10 — Decline a commitment
  1. Member submits a commit; you decline.
  2. Run CHECK.
✅ trip_commit · action = decline · actor = host.

5Part 3 — Member actions do as a member

Do these on a trip where you’re a member (Ohad is a member of Eyal’s trips, and vice-versa). The member is the one logged.

M1 — Edit your personal packing list
  1. As a member, toggle/edit your own personal packing items.
  2. Run CHECK.
✅ trip_personal_gear · trip set · actor = member. (no action)
M2 — Claim a gear item
  1. Claim a group-gear item (“I’ll bring it”).
  2. Run CHECK.
✅ trip_gear_claim · action = claim · trip = — (this event has no trip id; properties.item_id is set instead).
M3 — Un-claim a gear item
  1. Un-claim the item from M2.
  2. Run CHECK.
✅ trip_gear_claim · action = unclaim.
M4 — Propose a gear item
  1. As a member, propose a new gear item to the host.
  2. Run CHECK. (Then the host answers it in H7/H8.)
✅ trip_gear_request · action = request · trip set · actor = member.
M5 — Commit to the trip
  1. As a member, tap Commit to open the sheet.
  2. Select at least one commitment item — the Submit button stays disabled (greyed out) until you pick one.
  3. Tap Submit. If it fails you’ll see a “Could not submit” alert.
  4. Run CHECK. (Then the host answers it in H9/H10.)
✅ trip_commit · action = submit · trip set · actor = member.
⚠️ Tapping Submit with nothing selected does nothing (button disabled) → no event. That’s the most common “didn’t work”.
M6 — Share a trip invite host or member
  1. Tap “share / invite” on a trip (works from the trip detail and the just-published screen).
  2. Run CHECK.
✅ trip_invite_shared · trip set · actor = whoever shared. (no action)
