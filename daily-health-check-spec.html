Daily Deep Health Check 🩺

Check every core piece of the app on a schedule — before a real user hits something broken.

📅 2026-06-03 👤 Ohad + Claude 📊 for ~500 weekly users 🟡 Draft for review

The goal

Catch outages proactively. This is the “is the system alive?” layer.

Health Check = acts on its own, every day.  ·  Sentry = reacts when a real user hits an error. Two different jobs, working side by side.

Strategy — test building blocks, not paths

The 12 user paths share only ~9 building blocks. We test each block once instead of 12 overlapping tests. If a block is down, we instantly know which paths it takes with it.

How it’s wired

🤖 UptimeRobot Lives outside Supabase. Schedules the run, and sends you email + phone push on failure.
→
⚙️ health-check edge fn The brain. Runs all checks in parallel, returns a JSON report + HTTP 200 (ok) or 503 (fail).
→
🧱 The building blocks DB · Auth · OpenAI · S3 · Storage · Realtime · Google · Expo
✅
Why UptimeRobot owns the schedule: it’s outside Supabase, so if all of Supabase goes down (the function too), it still notices “no healthy reply” and alerts you. An internal cron can’t warn you that its own platform is dead.
⚠️
Deviation to bless: you said “Supabase cron.” The spec uses UptimeRobot as the scheduler instead (more robust + free push/email), and keeps pg_cron as optional Phase 2. Flip it if you prefer.

The health-check function

Returns

  • JSON report with per-check results + timings
  • HTTP 200 when all critical checks pass
  • HTTP 503 when any critical check fails
  • UptimeRobot alerts on: not 200, or body missing "ok":true

Security

  • Needs a secret token (x-healthcheck-token)
  • Stored as an edge-function secret
  • No token → 401
  • Blocks abuse + fake “healthy” responses

The permanent test user — one pre-made demo-healthcheck@swellyo.test (the demo* email auto-flags it as a demo user):

  • is_demo_user = true → invisible to matching & analytics
  • finished_onboarding = true → stops the hourly abandonment-reminder push from targeting it
  • No push token → no push can physically fire
  • Daily checks read/write this existing user — they never create or delete users (that’s fragile)

Signup-creation test WEEKLY · not daily

Verifies the user-creation path using Option 3 — create directly as demo:

  • Create auth user demo-signup-test-<ts>@swellyo.test
  • Insert users + surfers rows: demo = true, onboarding done, no push token
  • Assert: surfer row exists, demo auto-flagged, age computed by the trigger
  • Delete via auth.admin.deleteUser (cascades clean up)
⚠️
Honest gap: real signup builds the profile rows from client code, not an auth.users trigger. So this verifies “auth creation + surfer triggers fire,” not the exact client flow. Runs weekly because create/delete is the fragile part.

Error handling

  • Each check has its own try/catch + timeout
  • One failing check never aborts the others
  • Critical vs non-critical: only critical fails flip to 503
  • So a flaky Google quota blip won’t page you at 3am

Out of scope (YAGNI)

  • Full Google / Apple OAuth flow
  • Real push delivery to a device
  • pg_cron internal scheduling (maybe Phase 2)
  • A custom dashboard
  • Paid Sentry Log Drains

Done when…

  • The health-check function returns the JSON report + correct status
  • The permanent demo health-check user exists, flagged + onboarding-complete
  • UptimeRobot pings it and alerts via email + phone push on failure
  • Phase 1 checks pass on production — and a deliberately broken key produces a 503 + alert
