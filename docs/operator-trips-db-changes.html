Operator Trips — Database Changes

Everything the operator product adds or changes in the database. Updated 23 July 2026.

Rule 1: an operator trip is a group_trips row with hosting_style = 'C'. No separate operator tables. We add a few columns and new child tables. Trips A and B don't change at all.
Rule 2 (decided 23 Jul): "done" state is never stored. It is always read from the evidence: the document row, the agreement row, the medical form, the money ledger. There is no per-traveler state table.

New tables

group_trip_requirementsNEWwhat each trip asks travelers to do — one row per item
id / trip_iduuidBelongs to one trip.
kindtextpassport · waiver · medical · insurance · visa · flights · custom
req_typetextupload · acknowledge · pay
timingtextmust_have (no skip) · skippable (skip until a date)
title, help_texttextWhat the traveler sees.
deadline_days_beforeint"30 days before departure". Real date is computed, never stored — moving the trip keeps deadlines correct.
sort_order, is_activeint, boolOrder; removing an item just turns it off (history kept).
DECIDED 23 JULNo state table. Where each traveler stands is always computed:
upload items—No document row = not done. Row without approval = waiting. Approved row = done. Reject deletes the row.
waiver / "I agree"—Agreement row exists (for the current version) = done.
medical—Form's completed_at is set = done.
pay—The ledger adds up = done. (payment timing and method will be decided later on)
overdue / skip—Overdue = not done + deadline passed (computed). Skip writes nothing.
One SQL function does these joins and hands the app a clean list. Nothing can ever disagree with the evidence, because there is no second copy.

approved_at, approved_by—Empty = waiting for review. Set = approved. Reject = the row is deleted and the item re-opens.

This row IS the requirement's state for uploads — see "No state table" above.

Not tables, but part of the change

  • group-trip-documents — new private storage bucket. Files deleted within 30 days after the trip ends; a daily job does it.
  • is_trip_participant() — one new helper function for access rules. The existing is_trip_host() is only called, never modified.
  • New notification types (requirement added, rejected, overdue…) added to the existing enum — reuses the live push queue.
  • Two small read views: requirements with real due dates, and medical counts for the dashboard.
  • Feed RPCs (explore_feed etc.) must return any new display column, or the app never sees it.

Still to decide — these DO affect the schema

  1. Everything payment. The deposit column and the whole ledger table are sketches. Nothing is created until this is settled. (payment timing and method will be decided later on)
  2. Medical retention. The 30-day rule covers files, not rows. If medical rows also get deleted, that adds a purge job and maybe a column.
  3. Read / download logging. If we log who opened documents or medical data, that's one or two extra log tables.
  4. Hard C-only guard. Optionally a trigger so these tables can only get rows on hosting_style='C' trips. Not needed for v1.

Decided 23 July

  • ❶ No state table. "Done" is always derived from the evidence tables — group_trip_requirement_states was cut from the plan.
  • ❷ Visibility reuses the existing column — new 'link_only' value, no second column.
  • ❸ Nothing is added to the DB yet. Deposit column waits for the payments decision; when it comes it's one USD amount reusing budget_currency + frozen FX.
  • ❹ price_snapshot = column on group_trip_participants, shipped with the payments migration.
  • ❺ Names: group_trip_medical_forms (plural, like every child table), group_trip_acknowledgements (generic, waiver included), no doc_type on documents, all user FKs point at auth.users.

The other open questions (traveler notifications on edits, waiver re-agree, reminder cadence, approval of the waiver, what the dashboard shows after purge) change behavior, not tables.

Full SQL, access rules and flows live in docs/specs/operator-trips/. Migrations are applied by hand in the Supabase SQL editor.
