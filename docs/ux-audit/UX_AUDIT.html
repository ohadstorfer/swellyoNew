The 6 systemic themes

These patterns repeat across every domain — fixing the pattern is higher-leverage than fixing individual screens.

  1. Silent failure / “success on failure.” The biggest theme. Errors are catch→console.error→continue: delete-account, report, block, avatar upload, onboarding step 7, matching, privacy toggle. Users are told things worked when they didn't. → Gate every success UI on a real success response; on failure show a friendly, retryable error (friendlyErrorMessage already exists).
  2. Loading = a text swap or nothing, never a proper indicator. CTAs say “Signing in…” instead of a spinner; awaited actions sit dead; chats/profiles/blocked-list open blank; cold caches render false empty states. → Spinner-in-button convention + skeletons + distinguish empty/loading/error.
  3. No offline / connection feedback. DM, group, provider, and matching all track realtime health and reconnect, but never surface it; no global offline banner. → “Connecting…” pill in chat + global NetInfo strip.
  4. No global crash boundary. PostHogErrorBoundary re-throws everything else → any render crash = permanent white screen. → Top-level branded “Something went wrong — Reload”.
  5. Font weights render wrong on iOS at ~700 call sites (728 bare fontFamily+fontWeight vs 316 ff()). Headings/buttons ship thinner than Figma. → Sweep to ff().
  6. Theming & primitive drift. Button/Input unused; “dark” spelled 4+ ways; two reds, two greens; 282 blocking Alert.alert and no toast primitive. → Real <Button variant> + tokens + toast.

🔵 Low — nitpicks & cleanup

Representative sample (full list in per-domain files): aggressive activeOpacity={0.2} on conversation rows · near-limit character counter missing · hardcoded “10:45” timestamp in the team-welcome chat · “3 More” button label hardcoded regardless of count · dead styles shipping (opacity:0 “REMOVED” board container, skipButton opacity:0) · clipped Delete-account back button (fontSize:16 / lineHeight:15) · GoogleSignInTest.tsx / handleLogin dead stubs · hardcoded dimensions on match cards · duplicate TripPublished share-cancel string matching.

Suggested sequencing

  1. Stop lying to users (1 day): the 🔴 silent-failure set — delete-account, report+block, onboarding step-7, matching-error-vs-empty. Pure catch → gate-on-success + friendly retry. Highest trust impact, lowest effort.
  2. Stop blank/dead screens (1–2 days): global crash boundary (#5), chat cold-load skeletons (#8), trip-detail error/retry (#14), “Create Profile” spinner + inline errors (#20).
  3. WhatsApp table-stakes in chat (2–3 days): day/date separators (#6), offline/“Connecting…” pill (#7), group send-ticks + who's-typing, failed-voice retry, drop the Spanish strings.
  4. Trip CTA latency (1–2 days): make leave / remove / cancel / complete / member-mgmt / surftrips optimistic (the join/withdraw handlers are the working template).
  5. Matching payoff (1–2 days): real zero-match CTA, quick-reply chips, richer match cards + Message button, kill the silent 30s timeout.
  6. Systemic polish (ongoing): font sweep to ff(), per-screen StatusBar, toast primitive, <Button>/tokens, pinch-zoom viewer, NetInfo offline banner, linking config.
Worth remembering: a lot is genuinely above the bar already — optimistic send + upload-first media + reactions in DMs, the explore deck, the commitment flow, BottomSheetShell, FullscreenVideoPlayer, and friendlyError. The gaps cluster in error / empty / loading states and host-management paths, not core plumbing.
