Messaging System — Deep Review

A plain-English health check of chat: speed, crashes, refresh, heavy traffic, and long conversations. Reviewed by 5 focused agents · June 2026.

Needs attention Worth fixing Minor / polish Already good

The short version

  • Overall it's in good shape. The hard parts — caching, no duplicate chats, clean reconnects — are already done well.
  • Biggest single win: the chat list shows every conversation at once instead of only the visible ones. This gets slow for people with many chats.
  • One bug can freeze the chat: if sending a message times out, the screen can get stuck loading until you close and reopen it.
  • No safety net: if one message fails to draw, the whole app can go to a blank screen with no "try again".
  • Heavy traffic is mostly handled, but a few places do many small database calls at once that could be one.
⏱️ Load times

How fast chats open and how quickly the list appears.

GoodOpening feels instant
  • The chat list shows saved data immediately, then quietly refreshes in the background.
  • If the saved data is fresh (under 5 min), it skips the refresh entirely.
  • The top 10 chats are pre-loaded so they open with no wait.
  • A skeleton placeholder shows instead of a blank screen.
Needs attentionRefresh does many small database calls at once

Every time the app comes back to the foreground or a message arrives, it asks the database one question per chat — up to three questions each. For someone with 50 chats that's a burst of calls that should be a single one.

Fix: Combine them into one database call that returns everything at once.

Worth fixingTwo login checks back-to-back

Before loading chats, the app verifies who you are twice in a row, adding a small delay on every load.

Fix: Use the already-known login info instead of asking again.

MinorStarting a new chat re-fetches the whole list

Tapping someone to start a chat reloads all conversations just to check if one already exists — that info is already on the screen.

Fix: Check the chats already in memory first.

MinorPre-loading top chats one at a time

The top 10 chats are warmed up one after another instead of together, so a slow one holds up the rest.

Fix: Warm them all up in parallel.

💥 Crashes & stability

Where the chat can break, freeze, or show errors.

GoodSolid foundations
  • Failed messages stay visible with a "retry" instead of vanishing.
  • Network drops are caught and re-synced automatically.
  • Logout cleanly tears everything down in the right order.
  • Duplicate messages are well guarded against.
Needs attentionSend timeout can freeze the chat

If sending a message takes too long, the error is thrown in a way the app can't catch. The result: the loading spinner never stops and the send button stays disabled until you close and reopen the chat.

Fix: Handle the timeout the normal way so the screen recovers on its own.

Needs attentionNo safety net around the chat screens

If a single message fails to draw (e.g. odd image data), there's nothing to catch it — the whole app can go to a blank white screen with no way back.

Fix: Wrap the chat screens in a "Something went wrong — tap to retry" guard.

Needs attentionTrip "approve / decline" errors aren't caught

Approving or declining a trip commitment inside a chat can fail silently or, in worse cases, crash, because the error isn't handled.

Fix: Catch the error and show a simple "Couldn't do that" message.

MinorRare stale data after a delete

If a message is deleted at the exact moment another device edits it, slightly outdated info can get saved locally. Not a crash.

Fix: Skip the update when the message no longer exists.

🎯 Targeting & refresh

How a new message reaches the right chat and how the list stays up to date.

GoodRouting is clean
  • One channel per chat, reused — no duplicates.
  • New messages always land in the correct conversation.
  • The list moves the latest chat to the top instantly.
  • Unread counts get re-checked against the database to stay accurate.
Needs attentionA "lost connection" bug fix is built but not shipped

When you leave the last open chat, the live connection can briefly drop, and messages arriving in that gap are missed. The fix is already written but hasn't been released yet.

Fix: Ship the connection fix that's already on the branch.

Worth fixingNo pull-to-refresh on the chat list

If a message is ever missed, there's no way to manually refresh — you have to background and reopen the app.

Fix: Add the standard pull-down-to-refresh gesture.

Worth fixingReturning to the app resets your scroll spot

Coming back to the app reloads and re-sorts the whole list, which can jump you away from where you were scrolled.

Fix: Do a lighter "what changed" sync instead of a full reload.

MinorUnread count briefly counts twice

A new message can bump the unread badge by an extra one for a fraction of a second before it self-corrects.

Fix: Let only the authoritative re-check set the number.

🚦 Heavy traffic (many typing & sending at once)

How the system holds up when lots of people are active together.

GoodBuilt to scale
  • A busy person with 200 chats uses about 4 live connections, not 200.
  • Typing signals are throttled so they don't flood.
  • "Who's online" tracking is capped and efficient.
  • Rapid-fire sends can't create duplicates on the server.
Worth fixingDouble-tap can send twice (briefly)

The "don't send twice" lock turns on a moment too late, so a fast double-tap can slip a second send through before it engages. The server prevents true duplicates, but the screen can flicker an extra bubble.

Fix: Lock the send button the instant it's tapped.

Worth fixingWhole message list redraws on small changes

When a read receipt or similar small change happens, every visible message bubble redraws instead of just the affected one — causes lag during busy chats on older phones.

Fix: Make each bubble redraw only when its own data changes.

Worth fixingBursts trigger many database calls

When a burst of messages arrives, the app recalculates unread counts with two database calls per chat — for 50 chats that's 100 calls at once.

Fix: One database call that returns all the counts together.

Minor"Stopped typing" signal isn't throttled

Start-typing is rate-limited but stop-typing isn't, so busy chats send a bit more signal traffic than needed.

Fix: Apply the same throttle to the stop signal.

📜 Long conversations & big accounts

Chats with thousands of messages, and accounts with many chats.

GoodLong chats are handled well
  • Messages load in pages as you scroll up, not all at once.
  • The message screen only renders what's on screen.
  • Photos/videos load lazily with quick thumbnails first.
  • Saved history is capped on disk so storage stays small.
Needs attentionThe chat LIST renders every chat at once

Unlike the message screen, the conversation list draws all chats together instead of only the visible ones. With 85+ chats this is the single heaviest load on the screen — each row has an avatar, animations, and a badge.

Fix: Switch the list to a virtualized list that only draws visible rows.

Worth fixingOpen chats keep growing in memory

Scrolling far up in a long chat keeps every loaded message in memory for the session — fine for text, heavier for image-rich chats on old devices.

Fix: Keep only the most recent ~200 in memory and reload older ones if needed.

Worth fixingMessage screen keeps a wide render window

The chat keeps a large buffer of off-screen messages rendered, which adds memory pressure on image-heavy threads.

Fix: Reduce the render window size.

MinorCache cleanup reads everything one by one

Tidying up saved chats reads each saved chat separately to total up size, which briefly works the phone harder than needed.

Fix: Keep a small running total instead of recounting.
