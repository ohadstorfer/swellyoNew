Messaging System — Deep Review

Plain-English summary of how chat behaves today: speed, crashes, refresh, heavy traffic, and long conversations. Each point was found and then independently double-checked against the real code.

🚩 Top 3 things to fix

One bad message can crash the whole app

Fix soon

There is no safety net around the chat screens. If a single message fails to draw (a broken image link, a corrupt saved message, a missing field), the entire app goes to a blank screen instead of just showing a small error. You'd get a crash report, but the user sees a full crash.

Every incoming message does a burst of database work

Fix soon

While a chat is open, each single message that arrives makes the phone do ~5 database actions (mark-as-read, recount the unread badge, etc.) with no batching. In a busy group this multiplies by every person watching, and becomes the biggest source of load. It works now because traffic is low, but it scales badly.

Long chats slowly eat memory and get sluggish

Fix soon

As you scroll up through an old conversation, the app keeps every message you've loaded in active memory and never lets the old ones go. In a chat with thousands of messages this gradually slows the phone and can make scrolling janky. Opening a chat stays fast — the slowdown only builds up as you scroll or jump within a session.

⏱️ Loading times

Working well

The chat list opens instantly

Working well

Your chat list shows immediately from a saved copy on the phone, then quietly checks the server for anything new. If the saved copy is under 5 minutes old it skips the check entirely — so repeat opens feel instant and save data.

Opening a chat feels instant

Working well

Recent chats are kept ready in fast memory and shown with zero wait, then a tiny background fetch fills in anything you missed. The app even pre-loads your top 10 conversations right after the list appears, so tapping them opens with no delay.

Loading placeholders only show on the very first run

Working well

The grey placeholder rows appear only the first time, when nothing is saved yet. Every later visit shows the real list immediately — the right behavior.

Worth improving

The chat list draws every row at once

Worth improving

The list screen builds every conversation row up front instead of only the ones on screen. Fine for most people; for power users with dozens or hundreds of chats it makes the list heavier and less smooth. (The inside-a-chat screen already does this the efficient way.)

The background refresh does several waves of server calls

Worth improving

Refreshing the chat list from the server isn't one request — it's several stacked steps (who you chat with, the chats, last messages, names, unread counts). They're grouped to run in parallel, but on a slow connection a first-ever load or a manual refresh still takes a noticeable moment. This never affects the instant cached display.

Reconnect refresh does extra duplicate counting

Worth improving

After a reconnect, the app refreshes the affected chats one-by-one and then recomputes the unread badges a second time even though it just calculated them. It's hidden behind a single screen update so there's no flicker, but it's wasted work that adds up during busy bursts.

💥 Crashes & errors

Working well

Sending and uploading fail softly, never crash

Working well

Every outgoing action — text, photos, video, voice, edits, deletes — is wrapped safely. On failure it shows an alert and either undoes the change or marks the message "failed" with a retry, instead of crashing.

Camera, photo and video features degrade instead of crashing

Working well

The risky phone features (image compression, video thumbnails, cropping, the photo picker) are all guarded so they quietly do nothing when unavailable, rather than taking the app down. Applied consistently.

Worth improving

No safety net around the chat screens

Fix soon

See the top of this report: a single render error anywhere in messaging crashes the whole app to a blank screen instead of showing a small, recoverable error on just that screen.

One keyboard feature isn't guarded for the test environment

Worth improving

The chat screens use a keyboard-animation feature without the usual guard the rest of the app applies for the in-house test setup (Expo Go). It likely degrades gracefully, but since the team tests there, it's worth confirming on a real run.

🔄 Targeting & refresh (getting updates to the right chat)

Working well

One clean connection per open chat

Working well

Opening a chat uses a single live connection reused for everything — new messages, edits, deletes, typing dots, and "seen" ticks. It never opens a duplicate. This part is solid.

Missed messages are recovered after a dropped connection

Working well

The live pipe doesn't resend what you missed while offline, so the app deliberately asks the server for anything newer whenever you open a chat or reconnect. Missed messages get filled in correctly.

The known "list stops updating" bug is already fixed

Working well

The old method silently stopped delivering live updates for users with many conversations. The team already switched the default to a more reliable delivery method that doesn't have this problem.

Worth improving

Very large offline gaps fill in only 50 messages at a time

Worth improving

The catch-up after reconnect pulls up to 50 messages. If someone was offline through an unusually huge burst, the rest is only filled in by scrolling up. The code itself flags this limit — a rare edge case, not data loss.

A single chat's connection doesn't self-heal on its own

Worth improving

If one chat's live connection errors out, the service drops it and relies on the open screen to notice and reconnect, rather than fixing itself. It's wired and recovers, but recovery leans on the screen being open.

👥 Many people typing & sending at once

Working well

Duplicate sends are reliably prevented

Working well

Retries after a flaky network or an app restart mid-send can't create a second copy. Each message carries a unique stamp the database refuses to duplicate, and a persistent "outbox" safely re-sends unconfirmed ones. Solid and scale-safe.

Your own messages never double up or jump around

Working well

Your message appears instantly, then is swapped for the confirmed version — even when the live broadcast of your own message beats your send response. This is the tricky part of chat apps and it's handled carefully.

"Typing…" signals can't flood the network

Working well

The typing indicator is capped to at most one signal every half-second per chat, with an automatic "stopped typing" after a few idle seconds. Even a room full of people typing won't create a runaway stream.

The big scaling move is already done

Working well

The system already moved to the modern delivery method where the server sends one targeted event per channel and checks permission once when you join — instead of re-checking on every change for every person. This is the right foundation for growth.

Worth improving

Each incoming message triggers a burst of database work

Fix soon

See the top of this report — the system's biggest load multiplier. In a fast group, every active viewer does several database actions per message, with no batching.

Sender name/photo is looked up fresh for every message

Worth improving

For each arriving message the app re-fetches the sender's name and picture from the server, with no short-term memory. In a rapid burst the same person's details get fetched over and over — small queries, but wasteful at volume.

Big-group sends cost grows with group size

Worth improving

Each message notifies every group member one-by-one on the database side. Fine for small groups; the per-message cost climbs steadily as groups get large.

An extra "last updated" write on every single send

Worth improving

Every send also stamps the conversation's "last updated" time and waits for that write. In a fast group, many people stamping the same record at once compete for it and add a little delay to each send. The info is somewhat redundant with the message's own timestamp.

📜 Long conversations

Working well

Opening a long chat stays fast

Working well

Opening a years-old chat does NOT load thousands of messages. It shows an instant cached batch, then fetches only what's newer (usually nothing). A cold open with no cache loads just 30 messages. Speed doesn't depend on how old the chat is.

Loading a page of history is efficient

Working well

When a batch of messages loads, all the senders' names and photos are fetched in one combined request instead of one per message — so loading older history stays cheap even with many people in the thread.

Worth improving

Scrolled-up messages are never released from memory

Fix soon

See the top of this report. The more you scroll up in a long chat, the more the phone holds at once, and it's never trimmed for the rest of the session.

The message list keeps too much alive at once

Fix soon

The chat is set to keep a very large number of message bubbles active and doesn't release off-screen ones from view. Combined with the memory issue above, scrolling a long, media-heavy chat gets janky and stays memory-heavy.

Each update redraws the whole history

Worth improving

Every new message, edit, reaction, or "seen" tick rebuilds the full message list and a full lookup of all senders. Nothing in a short chat; in a chat holding thousands of messages it means several full passes per update, compounding the slowdown.

Tapping an old quoted reply can load a lot at once

Worth improving

Tapping a reply that points far back pages in older messages several times to find it — and (because of the memory issue) all of it stays loaded afterward. Repeated reply-jumps in a long thread are a fast way to balloon memory.

Bubbles redraw more than needed

Worth improving

Visible message bubbles aren't told to skip redrawing when their content hasn't changed, so unrelated updates (a typing dot, a reaction) redraw bubbles unnecessarily. Adds avoidable work in long, media-rich chats.
