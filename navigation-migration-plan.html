Navigation Migration Plan

From the hand-rolled boolean router in AppContent.tsx to a real card-stack system (react-navigation v7) — the same architecture Bluesky, WhatsApp and Instagram use. Based on 6 research streams + a full codebase audit.

2Rules of the system

  • Cards stack infinitely — trip → host profile → another trip → chat. Back always returns to the exact previous card, scroll position intact (the stack keeps covered screens alive for free — this replaces all our display:'none' hacks).
  • Sheets are cards too — notifications panel, filters: they're in back history, Android back closes them. Open a trip from the notifications panel → back returns to the open panel. (This fixes the original bug.)
  • Roots keep internal state — switch Lineup → Trips → back to Lineup: same scroll, same toggle pane. Re-tap the current root: pop to top + scroll to top (free, platform convention).
  • All 6 "where did you come from" flags get deleted — profileFromTripDetail, profileFromWelcomeOverlay, etc. The deck IS the memory.
  • Push notifications deep-link by pushing the right card; back pops home. Realtime stays in MessagingProvider above navigation (already correct today).

4Known landmines (from the war-stories research — all have fixes)

Never the big-bang switch. Teams that flag-flipped between old and new routers hit a silent state-corruption bug and took 2–4× longer. We go root-first, screen by screen — each phase independently shippable.
push() not navigate(). navigate() reuses an existing card — breaks profile → trip → profile chains and has a crash race on iOS. Every card open uses push().
iOS swipe-back vs carousels. The explore deck's horizontal swipe will fight the new edge-swipe-back gesture. Known fix (gesture coordination), tested in phase 2.
Android needs a 1-line native patch before the next store build (screen-fragment restore crash on RN 0.80+), and physical-Android testing for sheets — these bugs don't show on iOS.
Stay on react-navigation v7. v8 needs Expo SDK 55 — revisit after we upgrade.

5Coordination with Ohad

  • This rewires AppContent.tsx — the one file we never parallelize. While a phase is in flight, Ohad shouldn't land AppContent / TripsScreen changes without a heads-up.
  • Branch: nav-migration off main, one PR per phase into main, so his branch rebases cleanly after each phase.
  • His realtime + notification deep-link work is untouched in spirit — it actually gets simpler (the navigator replaces the pending-id plumbing).
  • Rule for both of us from day one: no new showX boolean flags. New screens use the navigator.
