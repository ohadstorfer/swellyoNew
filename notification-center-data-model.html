Notification Center — Data Model

Group-trip notifications. Designed to plug into your existing group_trip_* tables.

Shape: one row per recipient

One notifications row = one thing one person sees. We fan out on write (e.g. an admin update to 8 participants = 8 rows). Trips are small, so this is simpler than an event+recipients join and gives per-user read state for free.

The real decision (approved/denied) still lives on the source row via entity_type + entity_id. But each notification also has its own handled_at flag to mark the card done/dismissed — this works even for alerts that have no underlying request row. The notification carries a frozen render snapshot in data (names/qty/title) so the feed reads with no joins and old notices stay true even after names change.

Actionable cards (green) render their buttons against the live entity status — so a request already handled in another screen shows as resolved, no stale buttons.

Who creates the rows

Postgres triggers on the source tables (insert / status-change) fan out the rows. No client logic, fires no matter where the action came from (app, admin panel, SQL). The trigger reads names once and freezes them into data.

For admin notifications the trigger recipients = group_trips.host_id plus every group_trip_participants row with role = 'admin'. The fan-out already loops trip people, so including co-admins is one extra line — and no admin misses an alert.

Push notifications (planned) hook off the same trigger / a tiny after-insert step later — the table is the queue.

✅ Locked decisions (9 agent debates → judged)

#DecisionWhy (plain English)
1Fan-out: one row per personTrips are tiny, so one row per person is easy and needs less code.
2Add handled_at on the notificationMany alerts have no other row to check, so one done-flag per alert keeps it simple.
3Postgres triggers make the rowsThe app writes data in many places, so the database makes the note every time and none get lost.
4Freeze a snapshot in dataA notice is a record of what happened; if a name changes later the old notice still stays true.
5Keep audience column, show one feed for nowOne small saved label per row lets the database answer “what’s mine as host?” without messy client code.
6Per-row read_at onlyEach item already needs read_at, so use that for the badge too — no second number to sync.
7Flat rows, no group_keyTrips are tiny, so save one row each; add “3 joined” grouping later only if needed.
8Admin alerts → host (+ co-admins hook)Group trips only have host/member roles today — so this is host-only. The role='admin' hook is ready if co-admins are ever added.
9Keep forever, no cleanupThe app is tiny, so old rows cost almost nothing; deleting them now is work for no gain.
