🔔 Trip Notifications — Simple Testing Guide

Check every box on this page → the notification system is 100% working. Written in very simple English. Your progress is saved in this browser.

1What did we build?

When something happens in a group trip (someone joins, leaves, cancels…), two things happen:

🔔 The bell
A message appears inside the app, in the notification bell. This already worked before.
📬 The queue
NEW: a "push" (phone alert) is put in a waiting line in the database. A robot checks the line every minute and sends the alerts to phones.

3Your two magic check queries

You will use these two queries again and again. Run them in the Supabase dashboard → SQL Editor, or just ask Claude Code: "run check A" / "run check B".

CHECK A — the bell 🔔 (did the in-app message arrive, and for whom?)

select u.email as who,
       n.type,
       n.data->>'trip_title' as trip,
       n.data->>'stage'      as stage,
       to_char(n.created_at, 'DD/MM HH24:MI') as at_time,
       (n.read_at is not null) as read
from notifications n
join users u on u.id = n.recipient_id
where lower(u.email) in ('ohad.storfer@gmail.com', 'eyal@swellyo.com')
order by n.created_at desc
limit 8;

CHECK B — the push line 📬 (would a phone alert be sent, and what would it say?)

select u.email as who,
       q.type,
       q.status,
       q.skip_reason as reason,
       q.payload->>'title' as push_title,
       q.payload->>'body'  as push_text,
       to_char(q.send_after, 'DD/MM HH24:MI') as send_at,
       to_char(q.created_at, 'DD/MM HH24:MI') as at_time
from notification_queue q
join users u on u.id = q.recipient_id
order by q.created_at desc
limit 8;

How to read CHECK B results:

status / reasonMeaning
sent✅ PERFECT. The push was really sent — the phone should have buzzed with this title + text.
skipped / no_tokenThe receiver's phone never registered for pushes (or logged out). The bell message still works.
skipped / batched✅ Good. Several updates were grouped into one push (smart rule).
skipped / read_in_feed✅ Good. You already read it in the bell, so no push is needed (smart rule).
skipped / muted✅ Good. You muted this trip, so no push (smart rule).
pending⏳ Waiting. Check again in 1–2 minutes (or it is waiting for morning — see send_at).
skipped / shadowOld rows from before go-live (practice mode). New rows should never say this.

7Part 3 — Time tests (reminders) Uses SQL

Every morning at 06:07 UTC (09:07 in Israel), a daily robot looks at all active trips and their dates, and creates reminders. We will not wait days — we will change the dates of Trip 1 with SQL to trick the robot. This is safe: real users can't see trips.

Step 1 — find Trip 1's ID (run, then copy the id):

select id, title, start_date, end_date, status
from group_trips
where host_id = (select id from users where email = 'ohad.storfer@gmail.com')
order by created_at desc limit 5;

Step 2 — set a date, then run the robot. For each reminder below: run the small SQL (put your trip ID inside), then either ask Claude Code "trigger the reminder scan", or wait until after 09:07 the next morning. Then run CHECK A + B.

Each reminder fires only ONCE per trip. If you test the same one twice, no new row appears — that is by design (no spam). To re-test the same stage, ask Claude for a small cleanup SQL.
Reminder 1 — "Trip in 1 week"
update group_trips set start_date = current_date + 7, status = 'active'

✅ Bell: trip_reminder, stage week · Push line: title "[trip] — 1 week to go".

8Part 4 — Smart rules tests

The robot is smart. These tests prove the smart rules work.

Smart 1 — "You already saw it" rule

If you read the bell message before the push goes out, the push is cancelled. Normal events wait 60 seconds — that is your window.

  1. Eyal edits Trip 2's gear list (like Test 12).
  2. Within 1 minute, Ohad opens the app and opens the bell (so the message becomes "read").
  3. Run CHECK B.
✅ The row says skipped / read_in_feed — push cancelled because you already saw it. Smart!
Smart 2 — Grouping ("batching")

Many updates from the same trip in the same minute = ONE push, not many.

  1. Eyal quickly does 2–3 things on Trip 2 (edit gear list + post an update) within the same minute. Ohad does NOT open the bell.
  2. Wait 2 minutes. Run CHECK B.
✅ One row gets the push text, the others say skipped / batched. (Claude already saw this rule working once ✓.)
Smart 3 — Mute
  1. Ohad mutes Trip 2's chat in the app.
  2. Eyal posts an update on Trip 2.
  3. Wait 2 minutes. Run CHECK B. Then Ohad unmutes.
✅ The row says skipped / muted. The bell message still arrives (mute only stops pushes).
Smart 4 — Night time ("quiet hours") Uses SQL

Non-urgent pushes never wake people at night. They wait until 08:00 in the receiver's time zone. For this, the app must first save the time zone:

  1. Ohad opens the app from this branch (dev version) on his phone. It saves his time zone automatically.
  2. Check it worked:
select u.email, s.timezone
from surfers s join users u on u.id = s.user_id
where lower(u.email) in ('ohad.storfer@gmail.com', 'eyal@swellyo.com');
  1. In the evening (after 21:00 Israel time), Eyal does a normal event on Trip 2 (edits the gear list).
  2. Run CHECK B and look at the send_at column.
✅ send_at shows tomorrow morning 08:00 in Ohad's time zone (the result shows UTC, so for Israel it shows 05:00). The push politely waits for morning. Urgent ones (like "trip cancelled") do NOT wait — they go at once.
Bonus rule (now active): maximum 3 normal pushes per person per day. Number 4 and up get delayed 6 hours. Urgent pushes (decisions, cancellations) are never delayed. If a push seems "missing" during heavy testing, run CHECK B — it is probably pending with a later send_at. That is the rule working.
