✅ Done — deployed & verified Rate limiting on AI Edge Functions is per-isolate (ineffective)

DB: supabase/migrations/distributed_rate_limiter.sql · swelly-chat + swelly-trip-planning deployed 2026-06-10
Resolved 2026-06-10. Built a Postgres-backed distributed limiter: rate_limit_counters table + atomic check_rate_limit(bucket, max, window) function (fixed-window, INSERT…ON CONFLICT increment, bucket-scoped self-cleanup, search_path pinned, EXECUTE revoked from anon/authenticated). Notable: the drift check revealed live swelly-trip-planning had no limiter at all (was fully unprotected); it does now. Verified: limiter logic correct (3 allowed→4th denied), service_role can execute / anon+authenticated cannot, deployed code matches patches byte-for-byte, and both functions boot + return 401 on bad token (safe probe, no side effects). Fails OPEN on DB error so a hiccup never blocks real users. Repo files also updated (minimal limiter-only diffs) so future deploys keep it.
