Analytics Plan — Group Surf Trips

Metrics to add and what each needs. Planning doc — nothing built yet. Admin-gated, excludes demo & admins.

Ready in DB Needs JOIN New event Derived
01

Product health

  • Ready Trips created over time created_atA host publishes a new group trip.
  • Ready Trips by status — active / completed / cancelled statusCounts trips by their current state.
  • Ready Fill rate — count / capacity, % reaching full participant_count / max_participantsHow many of a trip's spots get taken vs its capacity.
  • Derived Cancellation rate cancelled / createdShare of trips a host cancels after creating them.
  • Derived Completion rate — of past-date trips completed vs end_dateShare of past trips that get marked completed.
  • JOIN Ghost trips — 0 requests / 0 members trips ⋈ requestsTrips that never got a single join request or member.
  • JOIN Time to first join & time to fill joined_at − created_atHow long a trip waits for its first member, and to fill up.
  • JOIN Lifecycle funnel — created → request → member → full → completed 4 tablesWhere trips drop off along the create-to-complete journey.
02

Supply

  • Ready Unique hosts DISTINCT host_idHow many different users have hosted at least one trip.
  • Ready Hosting style — A / B / C split hosting_styleSplit of hosts by the hosting style they pick.
  • Ready Budget distribution — tiers, ranges budget_tier, budget_min/maxHow trips spread across budget tiers and price ranges.
  • Ready Total spots offered SUM(max_participants)Total seats hosts have opened across all trips.
  • Ready Fixed vs flexible dates dates_set_in_stoneWhether hosts lock their dates or keep them flexible.
  • Derived New hosts over time MIN(created_at) per hostWhen each host ran their first-ever trip.
  • Derived Trips per host & repeat hosts COUNT per host_idHow many trips each host runs, and who hosts again.
  • JOIN Top destinations — country / locality destinationsWhich countries and places hosts choose most.
03

Demand

  • Ready Join requests created over time created_atA user requests to join a trip.
  • Ready Unique requesters DISTINCT requester_idHow many different users have asked to join any trip.
  • Ready Pending backlog & age status = 'pending'Requests still waiting on a host, and how long they've waited.
  • Derived Approval rate & decline / withdraw rate statusShare of requests hosts approve or decline, or users withdraw.
  • Derived Host response time reviewed_at − created_atHow fast a host answers a join request.
  • Derived Requests per trip COUNT per trip_idHow many join requests each trip pulls in.
  • JOIN Demand by destination / price requests ⋈ tripsWhich destinations and price points draw the most requests.
  • JOIN Funnel — request → joined → committed 3 tablesDrop-off from requesting to joining to committing.
04

Engagement & retention

  • Ready Members joined role = 'member'A user's request is approved and they become a member.
  • Ready Average crew size AVG(participant_count)Typical headcount on a trip.
  • Ready Commitments sent / approved commitment_statusA member sends a commitment and the host approves it.
  • Derived Commitment rate — committed / joined committedShare of joined members who go on to commit.
  • Derived Multi-trip users COUNT trips per userUsers who take part in more than one trip.
  • JOIN Trip chat activity conversation_id ⋈ messagesMessages sent inside a trip's group chat.
  • JOIN Post-join retention participants ⋈ app_openedWhether users keep opening the app after joining a trip.
05

Top of funnel

Not in the DB — needs new client events. Phase 2.

  • Event Explore impressions trip_card_viewedA trip card scrolls into view in Explore.
  • Event Card taps / detail opens trip_detail_openedA user taps a trip card to open its detail.
  • Event Create wizard drop-off — by step & style trip_create_step_nA user reaches a given step of the create-trip wizard.
  • Event Join intent vs submit join_request_startedA user taps Join, before actually submitting the request.
06

Phases

Phase 1 — all "Ready" + "Derived" metrics. RPCs + UI section, no client changes.
Phase 2 — top-of-funnel events (instrument client, no history).
Phase 3 — heavy JOINs: retention, chat, cohorts.

Reuses existing infra: admin gating, analytics-dashboard edge function, sparkline + funnel UI, date-range picker.
